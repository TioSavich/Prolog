# Fact-sheet: strategies

Written from [Stripped_Code/strategies/](../../Stripped_Code/strategies/) without reading any aspirational markdown. Cross-checked against Round 2 after drafting. Voice: SCENE register — hypothetical correspondences, not identities; productive failure is the point of contact.

## What the module defines

Six infrastructure files plus three families of content files (27 + 20 + 16), all under `strategies/`.

**Infrastructure.**

- `fsm_engine.pl` — a small recursive FSM driver. `run_fsm_loop/4` dispatches on `accept_state/1`; if not accepting, it calls `transition/3` (or `transition/4` under `run_fsm_loop_with_base/5`) and recurses, accumulating a history of `step(Name, Data, Interpretation)` entries. `run_strategy/4` wires a strategy module's `setup_strategy/5` and `extract_result_from_history/2` into this loop. The engine calls `incur_cost/1` from `formalization(grounded_arithmetic)` at entry and at each `emit_modal_signal/1`. It does not know anything about arithmetic; it only knows how to iterate a Prolog-as-FSM until `accept_state` succeeds.
- `hermeneutic_calculator.pl` — a flat dispatcher. `list_strategies/2` enumerates strategy names by operator (+ → COBO/Chunking/RMB/Rounding, - → eight subtraction strategies, \* → four, / → four). `calculate/6` pattern-matches on operator and strategy atom and delegates to the corresponding `run_*/N` predicate in the appropriate math/ module. No abstraction beyond dispatch. Does not appear in this module's own imports — it exists to be called from the learner.
- `strategies.pl` — a two-line namespace shim (`:- module(strategies, []).`) with no exports. Post-strip, this is all that remains; whatever registry role this file once played is no longer in the Prolog.
- `composition_engine.pl` — one exported predicate, `find_and_extract_copies/4`. Walks a tally list, selecting a unit type from a quantity list, with each step charging `incur_cost(unit_grouping)`. Used by fraction equivalence rules.
- `normalization.pl` — one predicate, `normalize/2`. Iteratively applies fraction equivalence rules (grouping, composition) via `fraction_semantics:apply_equivalence_rule/3` until fixed point, then `sort/2`.
- `math_benchmark.pl` — an I/O routine that runs canned integer, fractional, and nested-cognition examples through the grounded-arithmetic primitives and Jason's PFS. Prints `format` strings decorated with emoji. Not used by any other module in the sources examined; appears to be a demo harness.

**27 files in `math/`** (MANIFEST lines 66–92). The README calls 27 of them "strategy automata" by including the two base-counting PDAs in the count; by operator the breakdown is:

Each file exports a `run_*/N` entry point and (in the post-April-2026 conversions) FSM-engine hooks:

- Addition (5): `sar_add_counting_on`, `sar_add_cobo`, `sar_add_rmb`, `sar_add_chunking`, `sar_add_rounding`.
- Subtraction (9): `sar_sub_counting_back`, `sar_sub_cbbo_take_away`, `sar_sub_cobo_missing_addend`, `sar_sub_decomposition`, `sar_sub_sliding`, `sar_sub_chunking_a`, `sar_sub_chunking_b`, `sar_sub_chunking_c`, `sar_sub_rounding`.
- Multiplication (4): `smr_mult_c2c`, `smr_mult_cbo`, `smr_mult_commutative_reasoning`, `smr_mult_dr`.
- Division (4): `smr_div_cbo`, `smr_div_dealing_by_ones`, `smr_div_idp`, `smr_div_ucr`.
- Base-counting PDAs (2): `counting2`, `counting_on_back`.
- Fractions (3): `jason`, `jason_fsm`, `fraction_semantics`.

The pattern across the automata is uniform. An entry predicate converts integer inputs to `recollection([t,t,...])` via `formalization(grounded_arithmetic):integer_to_recollection/2`, constructs a `state(StateName, ...)` term, then either drives its own `run/3` recursion (the older pattern, visible in `sar_add_cobo`) or hands the state to `fsm_engine:run_fsm_with_base/5` (the newer pattern, visible in `smr_div_idp`). Transitions emit `s(comp_nec(...))` or `s(exp_poss(...))` modal signals into the PML engine at `arche_trace(incompatibility_semantics)`, and charge `incur_cost/1` at named granularities (`inference`, `state_change`, `modal_shift`, `slide_step`, `unit_count`, `fact_application`, etc.). The automaton terminates when it reaches a `q_accept` state, at which point the reversed history is returned as the trace.

**20 Indiana K-3 standards modules in `standards/`.** Each module corresponds to a specific Indiana standards cluster (K.NS.1 through 3.NS.5, plus CA clusters). Examples:

- `standard_1_ns_1.pl` exports `count_by_fives/3`, `count_backward_by_ones/3`, `count_backward_by_tens/3`, `teach_numerals_to_120/0`. Uses grounded-arithmetic primitives (`successor/2`, `predecessor/2`, `add_grounded/3`, `smaller_than/2`). Imports `standard_k_ns_1` and `standard_k_ns_2` as curricular prerequisites, which encodes the ordering of the progression in Prolog's import graph.
- `standard_2_ns_2_4.pl` exports `decompose_three_digit/4`, `compose_three_digit/4`, `expanded_form/2`, `describe_three_digit/2`. Depends on `standard_1_ns_2`'s two-digit predicates.
- `standard_3_ns_2.pl` exports `make_unit_fraction/2`, `iterate_unit_fraction/3`, `partition_whole/2`, `fraction_from_partition/3`, `valid_denominator/1`, `fraction_to_pair/3`. Uses a `fraction(Num, Den)` term structure built from recollections.

Standards modules charge `incur_cost(inference)` on public entry points but do not emit PML modal signals. They express curriculum progression as Prolog imports, not as an explicit elaboration graph.

**16 test suites in `tests/`** (plus `test_oracle_wiring.pl`). Each test suite is named for the standards module it covers. Per CLAUDE.md the aggregate count is 153 passing tests; the Prolog sources themselves do not record test counts.

## What axioms / inferences are asserted

What the code mechanically asserts, stripped of commentary:

- **An FSM accepts a strategy application.** For each math/ automaton M, `M:accept_state(State)` succeeds at exactly the terminal states the automaton was built to stop at. This is not a correctness theorem — it is a predicate. The relationship between "M:accept_state succeeds" and "the strategy applies to these operands" is not proved in the code; it is asserted by the author's construction.
- **Transitions charge cost.** Every transition in every automaton calls `incur_cost/1` from `formalization(grounded_arithmetic)`. The set of cost categories is open; each automaton introduces its own (`slide_step`, `unit_count`, `fact_application`, `completion`, `iteration`, `error_handling`, `equivalence_grouping`, `equivalence_composition`). There is no schema enforcing that these categories are commensurate.
- **Transitions emit PML modal signals.** Every non-trivial transition in the math/ automata calls `s(comp_nec(Atom))` or `s(exp_poss(Atom))` from `arche_trace(incompatibility_semantics)`. The atom is a descriptor of the transition (`focus_on_bases`, `shift_to_ones`, `applying_discovered_multiplication_fact`). What the PML engine does with these atoms is not the automaton's concern; the automaton asserts only that a modal shift occurred.
- **Hermeneutic dispatch is finite and flat.** `hermeneutic_calculator:calculate/6` has one clause per (operator, strategy-name) pair: 4 + 8 + 4 + 4 = 20 clauses. There is no fallback, no default, no similarity metric between strategies. A request for an unlisted strategy simply fails.
- **Curricular imports encode prerequisite order.** In `standards/`, `standard_1_ns_1` imports from `standard_k_ns_1` and `standard_k_ns_2`; `standard_2_ns_2_4` imports from `standard_1_ns_2`. The sequence is coded as a Prolog dependency, not named as a progression.
- **Fraction equivalence has two rules.** `fraction_semantics:apply_equivalence_rule/3` asserts two rules. Grouping: if a denominator-many copies of `unit(partitioned(D_Rec, ParentUnit))` appear in a quantity, they can be replaced with `ParentUnit`. Composition: a nested partitioned unit `unit(partitioned(A, unit(partitioned(B, P))))` can be flattened to `unit(partitioned(A*B, P))`. The first rule is cost-charged as `equivalence_grouping`; the second as `equivalence_composition`.
- **27 automata do not claim to enumerate children's arithmetic.** No file says "these are all the strategies." The list is what has been coded. Provenance (Carpenter/Fennema CGI via Hackenberg per CLAUDE.md) is not carried in the Prolog sources after comment-stripping — it survives in the module-level `math/README.md` but not in any `:- module(...)` or predicate declaration.
- **Standards modules assert curricular judgment as dispatch.** The claim that "strategy X is appropriate for standard Y" is encoded by which predicates a standards module exports and imports. Nothing in the code validates that judgment against curricular documents or against student behavior.

## What the module does NOT do

- **No elaboration graph between strategies.** Strategies are a flat list. The hermeneutic calculator dispatches by name; it does not know that COBO elaborates on counting-on, that RMB elaborates on COBO, or that sliding is a specialization of decomposition. The dependency-chain comments in `GROUNDING_STATUS.md` are not reflected in any Prolog relation the code could query.
- **No empirical validation.** The automata are formal models. The code does not compare traces to student performance data; there is no student-behavior corpus in the module; no automaton is annotated with "this pattern was observed in N children." Grounding, when discussed in the code's own documents, is the is/2-vs-grounded-arithmetic question, not empirical grounding against classroom observation.
- **No crisis integration for fractions.** `jason_fsm.pl` runs PFS and FCS as self-contained FSMs. `standard_3_ns_2.pl` models unit-fraction structure with a `fraction(Num, Den)` term. Neither connects to the ORR cycle in `learner/`; the meta-interpreter has no goal type for `fraction/2` or `fraction/3`. The `fraction` world (rationals, recollection-based unit structures) and the ORR world (Peano/integer terms dispatched through the oracle) are structurally incompatible.
- **No named progression across standards.** Curricular order is encoded as imports but there is no `progression(From, To)` relation, no `elaborates(StandardA, StandardB)` fact, no formal representation of how K.NS.1 builds toward 1.NBT.A.1. The progression exists in the grain of which modules import which; a reader can reconstruct it but the Prolog cannot.
- **No cost calibration.** `incur_cost(inference)` and `incur_cost(slide_step)` are strings that flow into whatever counter the learner's config provides. The strategies module does not specify what the cost multipliers should be; those live in `learner/config.pl`. Strategies declare their cost categories; they do not claim the categories are equal, ordered, or measurably distinct.
- **Cost accounting is partial (was vacuous pre-2026-04-16).** Before the fix, `formalization/grounded_arithmetic.pl:incur_cost/1` had body `true` — every automaton's `incur_cost(...)` call was a no-op. After the fix, direct (non-meta-interpreter) callers accumulate into a module-local counter in `grounded_arithmetic` readable via `direct_cost_accumulated/1`.
