# Arche-Trace to Learner: Where the Learner's Formalization Breaks

## The Claim

The arche-trace module marks three points where the learner model's formalization
goes hollow. At each point, the formal machinery produces a structural result but
the proof object contains no content. These are not bugs. They are the precise
boundaries where computation ends and interpretive judgment — by a teacher, an
LLM, or a reflective practitioner — must begin.

The three augmentation points are:
1. **Proof erasure** — the sequent calculus succeeds but the proof is hollow
2. **Resource exhaustion** — the prover runs out of budget, signaling that the
   current approach is too expensive
3. **Pathology detection** — the prover finds a cycle trapped in compression,
   signaling stagnation

Each point is a different kind of failure, and each calls for a different kind
of intervention.

## What Exists in Code

### The Learner's Execution Path

The learner module processes goals through this chain:

```
execution_handler:run_computation(Goal, Limit)
  -> meta_interpreter:solve(Goal, I_In, I_Out, Trace)     [solve/4 wraps solve/6]
  -> solve/6 dispatches:
       - true:           succeed
       - incur_cost(X):  charge cost, accumulate tension, check crisis
       - s(comp_nec(X)): switch to compressive context, charge modal_shift
       - object_level:add(A,B,R): try learned strategies (LIFO), then fall back
       - other goals:    check_norms, resolve via clause/2, recurse
  -> on perturbation:
       execution_handler:handle_perturbation classifies crisis
       -> efficiency_crisis:  consult oracle for faster strategy
       -> unknown_operation:  consult oracle for new operation
       -> normative_crisis:   expand domain (N -> Z -> Q)
       -> incoherence:        belief revision
       -> tension_instability: catastrophe surface unstable
```

Key files: `learner/execution_handler.pl`, `learner/meta_interpreter.pl`,
`learner/tension_dynamics.pl`, `learner/oracle_server.pl`, `learner/config.pl`.

### The Arche-Trace Machinery

The arche-trace module provides the formal justification layer:

**Scene-agnostic sequent engine** (`arche-trace/incompatibility_semantics.pl`):
- `proves/1`: takes `(Premises => Conclusions)`, succeeds if derivable
- Includes axiom sets from all modules via `:- include(...)`:
  - `formalization/axioms_robinson.pl`
  - `formalization/axioms_geometry.pl`
  - `formalization/axioms_number_theory.pl`
  - `pml/axioms_eml.pl`
  - `learner/axioms_domains.pl`
- Rule priority: identity > explosion > material axioms > forward chaining > structural rules

**Resource-tracked prover** (`arche-trace/embodied_prover.pl`):
- `proves/4`: `proves(Sequent, R_In, R_Out, Proof)`
- Tracks modal context (compressive/expansive) and cognitive resources
- Returns either `proof(RuleName, Sequent, SubProofs)` or `erasure(RuleName)`
- Throws `perturbation(resource_exhaustion)` when budget is insufficient

**Trace mechanism** (`arche-trace/automata.pl`):
- `generate_trace/1`: creates an attributed variable with `arche_trace` attribute
- `contains_trace/1`: detects whether any variable in a term carries the attribute
- `attr_unify_hook/2`: the Trace propagates to other variables on unification
  but FAILS when unified with any concrete term — it resists stabilization

**Critique module** (`arche-trace/critique.pl`):
- `reflect/2`: analyzes proof for pathology (bad infinite cycles)
- `accommodate/1`: applies sublation — stress recording, belief revision,
  conceptual elevation signal
- `stress/2`: dynamic predicate tracking accumulated stress per commitment

**Dialectical engine** (`arche-trace/dialectical_engine.pl`):
- `run_computation/2`: wraps `proves/4` in catch/throw for perturbation handling
- `run_fsm/4`: generic FSM executor

### The Three Augmentation Points

#### Point 1: Proof Erasure

When a sequent contains a term with the `arche_trace` attribute (generated by
`automata:generate_trace/1`), the embodied prover replaces the proof object
with `erasure(...)`.

The mechanism (`embodied_prover.pl:98-105`):

```prolog
construct_proof(RuleName, Sequent, SubProofs, Proof) :-
    ( member(erasure(_), SubProofs) ->
        Proof = erasure(propagation)         % child erased -> parent erased
    ; ( contains_trace(Sequent) ->
        Proof = erasure(RuleName)            % sequent has Trace -> erased here
      ; Proof = proof(RuleName, Sequent, SubProofs)  % normal proof
      )
    ).
```

Four documented erasure points (`arche-trace/ARCHE_TRACE_ERASURE.md`):

| Point | Sequent | Proof Result | Why It Erases |
|-------|---------|-------------|---------------|
| Identity with Trace | `[s(I_f)] => [s(I_f)]` | `erasure(identity)` | Self-identity of the I-Feeling succeeds structurally but produces no inspectable content |
| S-O Inversion | `[s(comp_nec(I_f))] => [o(exp_nec(I_f))]` | `erasure(propagation)` | Converting subjective experience to objective knowledge loses content |
| Double Negation | `[neg(neg(s(I_f)))] => [s(I_f)]` | `erasure(propagation)` | Law of excluded middle holds structurally but the proof is empty |
| Oobleck Transfer | `[s(comp_nec(I_f))] => [o(comp_nec(I_f))]` | `erasure(propagation)` | Transferring modality across person-positions with Trace present |

Normal proofs (without Trace) produce full `proof(...)` objects. The erasure
occurs precisely and only when the I-Feeling — subjective experience that
resists representation — enters the derivation.

#### Point 2: Resource Exhaustion

The embodied prover tracks a budget (`R_In`) decremented by each inference step.
Cost depends on modal context (`embodied_prover.pl:46-48`):

```prolog
get_inference_cost(compressive, 2).
get_inference_cost(expansive, 1).
```

When the budget is insufficient, `check_viability/2` throws
`perturbation(resource_exhaustion)`.

In the learner's meta-interpreter, a parallel mechanism exists: `solve/6` tracks
`I_In/I_Out` and calls `tension_dynamics:check_crisis`, which throws
`perturbation(tension_instability)` when the second derivative of tension goes
negative (stability < 0, computed over a 9-entry rolling window).

Both mechanisms signal the same thing: the current approach is too expensive to
continue. The difference is granularity — the embodied prover's budget is a
simple counter, while the tension dynamics model a catastrophe surface.

#### Point 3: Pathology Detection

`critique.pl:reflect/2` analyzes proof trees for cycles. A **bad infinite** is
a cycle where every node is compressive (`critique.pl:100-107`):

```prolog
is_bad_infinite(Cycle) :-
    Cycle \= [],
    forall(member(Node, Cycle), is_compressive_node(Node)).
```

`accommodate/1` handles three trigger types:
- **Resource exhaustion**: records stress, signals external intervention needed
- **Incoherence**: identifies most-stressed commitment, retracts it (belief revision)
- **Bad infinite**: marks cycle elements as stressed, signals need for conceptual
  elevation (sublation) — the system cannot auto-generate new concepts but
  indicates where they are needed

## Where the Claim Holds

### Erasure Works in Tests

The PML core tests (`pml/tests/core_test.pl`) verify erasure:

```prolog
% Test: identity with trace produces erasure
i_feeling(I_f),
proves([s(I_f)] => [s(I_f)], 50, _, Proof),
Proof = erasure(_).            % succeeds

% Test: identity without trace produces normal proof
proves([a] => [a], 10, _, Proof),
Proof \= erasure(_).           % succeeds — normal proof
```

The boundary is precise: the same derivation rule (identity) produces a real
proof for abstract terms and an erased proof for Trace-bearing terms.

### Crisis Classification Is Operational

The learner's `execution_handler.pl` classifies 5 crisis types and dispatches
to appropriate recovery:
- `efficiency_crisis` -> oracle consultation for a faster strategy
- `unknown_operation` -> oracle consultation for a new operation
- `normative_crisis` -> domain expansion
- `incoherence` -> belief revision
- `tension_instability` -> catastrophe-surface-triggered crisis

The oracle server (`learner/oracle_server.pl`) successfully dispatches to
strategy modules and returns black-box interpretations. The HTTP server
(`learner/server.pl`) exposes this via POST `/api/compute`.

### Domain Axioms Enforce Normative Boundaries

`learner/axioms_domains.pl` prohibits subtraction below zero and division below
the divisor in the natural number domain. When these prohibitions are violated,
the system throws `perturbation(normative_crisis(...))`, which the execution
handler routes to domain expansion. This is a working boundary: the learner
encounters a computation that its current normative frame cannot sanction.

## Where the Claim Exceeds What's Formalized

### The Learner Does Not Use the Arche-Trace Prover

This is the central gap — but it requires nuance. The learner DOES import
from the arche-trace layer:

- `meta_interpreter.pl` imports modal operators (`s/1`, `comp_nec/1`, etc.)
  and `check_norms/1` from `incompatibility_semantics.pl`
- `more_machine_learner.pl` imports `proves/1` to ground its foundational
  successor capability: `successor(X, Y) :- proves([] => [o(plus(X, 1, Y))])`

What the learner does NOT use is the resource-tracked embodied prover
(`proves/4`) or the dialectical engine (`run_computation/2`). The learner's
`solve/6` has its own inference tracking and perturbation handling, running
in parallel with the arche-trace machinery rather than through it.

The `SYSTEM_ASSESSMENT.md` argues this parallel is intentional: `solve/6`
models computing (cognition as embodied activity), while `proves/4` models
justifying (cognition as normative practice). Negarestani's distinction
between protosemantic navigation and cognitive navigation of logical space
maps onto the same split. The manuscript warns: merging them conflates what
the system can *do* with what it can *give reasons for*.

The arche-trace module and the learner module implement parallel versions of
the same cycle:

| Aspect | Learner (meta_interpreter) | Arche-Trace (embodied_prover) |
|--------|---------------------------|-------------------------------|
| Budget tracking | `I_In/I_Out` in solve/6 | `R_In/R_Out` in proves/4 |
| Modal context | `CtxIn/CtxOut` in solve/6 | `CtxIn/CtxOut` in proves_impl/7 |
| Crisis signal | `tension_dynamics:check_crisis` | `check_viability/2` throws perturbation |
| Proof objects | Execution trace (list of terms) | `proof(...)` or `erasure(...)` |
| Pathology | Not checked | `critique:reflect/2` |

The three augmentation points exist in arche-trace's machinery but are not
wired into the learner's execution path.

### Module Name Collision

`embodied_prover.pl` declares its module as `incompatibility_semantics`
(line 1), colliding with the scene-agnostic engine in
`arche-trace/incompatibility_semantics.pl`. The two files cannot be loaded
simultaneously. This is documented as a known issue in the spec.

### Critique Module Not Called

`critique.pl`'s `reflect/2` and `accommodate/1` are not invoked by the
learner's execution handler. The learner handles perturbations via its own
`handle_perturbation/5` in `execution_handler.pl`, which consults the oracle
directly rather than running pathology analysis on a proof object.

### Dialectical Engine Is Unused

`dialectical_engine.pl` provides `run_computation/2`, which wraps the embodied
prover in the same catch/throw pattern the learner uses. But the learner calls
`execution_handler:run_computation/2` (its own version), not the dialectical
engine's version.

### Erasure Has No Pedagogical Consequence

When the embodied prover returns `erasure(identity)`, nothing in the system
acts on that signal. The erasure marks where a teacher or LLM must supply
interpretive judgment, but no mechanism routes the erasure to an oracle
consultation or generates a prompt for the teacher. The erasure is a marker,
not a trigger.

## What Would Strengthen This Interface

### Option A: Wire the Arche-Trace Prover into the Learner

Replace or augment the learner's `solve/6` with calls to
`embodied_prover:proves/4` for the justification layer. The computing layer
(strategy execution) stays in the meta-interpreter. The justifying layer
(derivation and proof objects) routes through the embodied prover.

This would require:
- Resolving the module name collision (give `embodied_prover.pl` its own
  module name)
- Defining a handoff protocol: when does `solve/6` delegate to `proves/4`?
- Handling the case where `proves/4` returns `erasure(...)` — what does the
  learner do with a hollow proof?

### Option B: Document the Parallel as Intentional

If the two execution paths are genuinely modeling different things — the
meta-interpreter models *doing arithmetic* while the embodied prover models
*justifying arithmetic* — then the parallel is the point. The manuscript
distinguishes computing and justifying as two distinct activities.
`SYSTEM_ASSESSMENT.md` is explicit: "A child who can compute 8+5=13 is not
the same as a child who can justify why 8+5=13."

This option frames the interface as a philosophical correspondence rather
than a code connection: for every computation the learner performs, there
is a corresponding justification the prover could construct. The interesting
cases are where the justification erases — where the child can do the
arithmetic but the formal system cannot produce reasons for it.

The partial connections that already exist (modal operators, `check_norms`,
`proves/1` for successor grounding) are consistent with this framing: the
learner borrows the *vocabulary* of the justification layer without entering
its *proof regime*.

This option requires no code changes but demands a clear articulation of
what the correspondence is and where it breaks down.

### Option C: Route Erasure to the Oracle

When `proves/4` returns `erasure(RuleName)`, generate a structured query to
the oracle/teacher: "The derivation succeeded structurally but the proof is
hollow at [RuleName]. What interpretive account fills this gap?" This would
make erasure a trigger for pedagogical intervention, not just a marker.

### Regardless of Option

- Resolve the module name collision.
- Add a worked example: show a learner computation that, when justified through
  the embodied prover, hits an erasure point. Trace both paths (computing and
  justifying) for the same problem, showing where they diverge.
- Document which of Options A, B, or C the project commits to, so future
  development has a clear direction.
